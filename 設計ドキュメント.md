# ゲームトレンドダッシュボード 設計ドキュメント

作成日: 2026-07-15

## 1. 概要・目的

現在流行っているゲームを、客観的な指標(検索関心度・プレイ人口・配信人気度・売上/収益・SNS話題度)から可視化するダッシュボードアプリ。

主な狙い:
- 「なんとなく話題」ではなく、複数の客観指標を横断して"今どれくらい流行っているか"を定量的に把握する
- 基本無料(F2P)と買い切り型など、ビジネスモデルが異なるゲームを無理に同一基準で比較しない
- リアルタイムのスナップショットだけでなく、週次・月次の推移や勢いも見られるようにする

前提条件: **運用コストはゼロ**(有料API・有料データソースは使わない)

## 2. データソース一覧

### 2.1 検索関心度
| 指標 | 取得元 | 取得方法 | コスト | 備考 |
|---|---|---|---|---|
| 検索インタレスト指数 | Google Trends | pytrends(非公式) | 無料 | 相対指数(0-100)。**1リクエストにつき最大5キーワードまでしか比較できず、返る値はそのリクエスト内での相対値**。追跡ゲームが5を超える時点で単純な値の突き合わせができない。共通のアンカーキーワード(毎回同じキーワードを1つ含めて全バッチで基準化する)を使ったリスケーリングが必要。リクエスト頻度にも注意(叩きすぎるとIPブロックのリスク) |
| 急上昇関連クエリ | Google Trends | pytrends(非公式) | 無料 | Hot Words機能のベースとして使用 |

### 2.2 プレイ人口・エンゲージメント(買い切り系中心)
| 指標 | 取得元 | 取得方法 | コスト | 備考 |
|---|---|---|---|---|
| 同時接続数 | Steam | 公式API(`ISteamUserStats/GetNumberOfCurrentPlayers`) | 無料 | Steam版があるゲームのみ取得可能 |
| レビュー増加ペース | Steam | 公式API/ストアページ | 無料 | 売上の代理指標として使う(Boxleiter法的な考え方) |
| ディスカッション数 | Steam | コミュニティページ | 無料 | コミュニティ熱量の補助指標 |

### 2.3 配信人気度(カテゴリ共通)
| 指標 | 取得元 | 取得方法 | コスト | 備考 |
|---|---|---|---|---|
| 視聴者数・配信者数 | Twitch | 公式Helix API | 無料 | 買い切り/F2P問わず取得可能。カテゴリ横断比較の軸として重要 |

### 2.4 売上・収益(F2P系中心、代理指標)
| 指標 | 取得元 | 取得方法 | コスト | 備考 |
|---|---|---|---|---|
| 売上ランキング順位 | App Store / Google Play | 公式APIなし、ランキングページ参照 | 未定(取得方法要検討) | 実額は取得不可のため順位で代用。公式APIが存在しないため、取得方法(非公式ライブラリ/スクレイピングの是非)は8節の未決事項として別途検討する必要がある |
| レビュー増加ペース | App Store / Google Play | ストアページ | 無料 | 補助指標 |

### 2.5 SNS・コミュニティの盛り上がり(Hot Words)
| 指標 | 取得元 | 取得方法 | コスト | 備考 |
|---|---|---|---|---|
| 急上昇関連クエリ | Google Trends | pytrends | 無料 | 2.1と共通 |
| 新着動画数・再生数の伸び | YouTube | 公式Data API | 無料枠(1万ユニット/日) | 検索1回=100ユニット。1日100回程度の検索が可能 |
| 投稿数・コメント数 | Reddit | 公式API | 無料枠(個人規模) | 英語圏中心の偏りに注意。**2023年以降、新規OAuthアプリ登録にはReddit側の事前承認(申請)が必要**になっており、単純な自己登録では使えない可能性がある点に注意 |
| 記事本数の推移 | Google News RSS | RSS(無登録) | 無料 | 件数は少なめだが補助指標として有効 |

### 2.6 除外したデータソース
- **X(Twitter)**: 公式APIが有料($200/月〜)のため断念。スクレイピング代替もToSリスク・不安定さ(頻繁に破損)から不採用
- **売上本数(公式集計)**: 公式APIが存在せず自動収集不可のため見送り
- **Sensor Tower等の有料売上額データ**: コスト条件により見送り

## 3. ゲーム分類とマルチプラットフォーム対応

### 3.1 カテゴリ分類の考え方
「流行っている」の意味がビジネスモデルによって異なるため、**収益モデル(買い切り / F2P)でカテゴリ分け**し、カテゴリごとに主指標を変える。

| カテゴリ | コア指標 |
|---|---|
| 買い切り型 | Steam同時接続数、Steamレビュー増加ペース |
| 基本無料(F2P) | ストア売上ランキング順位、レビュー増加ペース |
| 共通(横断比較用) | Trends検索指数、Twitch視聴者数、YouTube動画数、Reddit投稿数 |

分類はプラットフォームではなく**そのゲームの主収益モデル**で判定する(例: 原神はSteam版があってもF2Pに分類)。

### 3.2 マルチプラットフォーム対応
ゼンレスゾーンゼロのように、モバイル・PC・Steamに同時展開されるゲームが存在する。この場合、**ゲームを単位として「取得できる指標をすべて集める」柔軟な構造**にする。

- Steam版がある: Steam同接数・レビュー増加ペースも追加で収集
- モバイル版がある: ストアランキングも収集
- 欠損指標は「その指標を保有するゲーム集団内」でパーセンタイル順位を計算し、加重平均の対象から自然に除外する
- ダッシュボード上では合成スコアだけでなく、**プラットフォーム別の内訳を必ず併記**し、透明性を担保する(例: Steam同接: 12,000人 / モバイル売上順位: 3位 / Twitch: 8,500人)
- 注記: Steam同接数は「Steamクライアント経由のみ」の数値であり、公式ランチャー版やモバイル版のプレイヤーは含まれない点を常にUI上に明示する

## 4. スコアリングロジック

### 4.1 正規化: パーセンタイル順位
Steam同接数やTwitch視聴者数は分布が偏りやすい(一部の大作が桁違いに大きい)ため、min-max正規化は外れ値に弱い。**その日時点でトラッキングしている全ゲーム内でのパーセンタイル順位(0-100)** に変換する方式を採用する(以降「週」「月」等の表記は全てこの日次パーセンタイルを集計したものを指す)。

なお、Trends指数はpytrendsの制約(2.1参照、1リクエスト最大5キーワード・バッチ内相対値)により、そのままでは日次パーセンタイルの母集団比較に使えない。共通アンカーキーワードで全バッチを同一スケールにリスケーリングした後の値を正規化対象とする。

### 4.2 「水準」と「勢い」を分離する
「流行っている」には2つの意味が混在しているため、スコアを分離して持つ。

- **水準スコア(Level)**: 現在の絶対的な人気度(パーセンタイル順位の加重平均)
- **勢いスコア(Momentum)**: 直近の伸び率(例: 直近7日平均 ÷ 過去28日平均)をパーセンタイル化したもの

原神のような定番大作は水準は高いが勢いはほぼゼロ、発売直後の新作は水準が低くても勢いが急上昇する、というケースを区別して捉えるため。

**コールドスタート対応**: 運用開始直後(過去28日分のデータが揃っていない期間)や追跡対象に新規追加したゲームは、分母データが存在せずゼロ除算・値の発散が起こる。この場合は勢いスコアを「算出不可(N/A)」として扱い、UI上も「データ収集中」等で明示する。十分なデータが蓄積された時点(目安28日経過後)から勢いスコアの算出を開始する。

### 4.3 二層スコア構造
| 層 | 目的 | 使う指標 |
|---|---|---|
| カテゴリ内コアスコア | 同業態内での比較 | 3.1のコア指標(カテゴリ別) |
| 横断注目度スコア(Attention Score) | 全ゲーム共通の話題性比較 | Trends、Twitch、YouTube、Reddit(カテゴリ共通指標のみ) |

各層で水準スコア・勢いスコアをそれぞれ算出し、最終的に「水準50% + 勢い50%」を目安に合成する(重みは調整可能なパラメータとして持たせる)。

### 4.4 算出フロー
1. 日次で各ソースから生データを収集・保存
2. 日次バッチで、その日時点の全ゲームに対し指標ごとにパーセンタイル順位を算出
3. 水準スコア(その日のパーセンタイル加重平均)と勢いスコア(直近7日 vs 過去28日の比率をパーセンタイル化)を算出
4. カテゴリ内コアスコア・横断注目度スコアをそれぞれ保存
5. Hot Words: Trends急上昇クエリ + YouTube動画タイトル + Redditタイトルから頻出語を抽出し、直近頻度 ÷ 過去平均頻度でスパイク検知(過去データが不足する期間は勢いスコアと同様にN/A扱いとし、算出しない)

## 5. 時間軸設計

### 5.1 データ収集頻度
- Steam同接数・Twitch視聴者数: 変動が大きいため1日数回サンプリング(例: 4時間毎)し、日次平均・最大値を保存
- Trends・YouTube・Reddit・ストアランキング・News RSS: 日次1回で十分

### 5.2 期間別の見せ方
| 期間 | 適した可視化 | 何がわかるか |
|---|---|---|
| リアルタイム/直近 | 4象限マップ、ランキング表 | 今この瞬間の状況 |
| 過去1週間 | 折れ線グラフ(日次推移) | 直近の勢いの中身、急上昇/急落のタイミング |
| 過去1ヶ月 | バンプチャート(週次ランキング変動) | 中期的な順位の入れ替わり |
| 過去1ヶ月(密度) | カレンダーヒートマップ | 話題が盛り上がった日がいつだったか |

期間セレクター(リアルタイム/7日/30日/カスタム)をダッシュボード上部に置き、選択に応じて表示を切り替える。期間比較の差分表示(今週 vs 先週の順位変動・スコア変化)もランキング表に組み込む。

データは常に日次スナップショットとして蓄積し、週次・月次の集計は表示時にオンデマンドで計算する(収集頻度と表示粒度を分離)。

## 6. システムアーキテクチャ(静的サイト + GitHub Actions構成)

常時稼働のサーバー・データベースは持たず、収集・スコアリングをGitHub Actions上のバッチ処理で完結させる。APIサーバー(FastAPI等)は不要で、フロントエンドは静的ホスティングされたJSONを直接fetchする。

```
[GitHub Actions: 収集ジョブ] -> data/raw/*.json -> [GitHub Actions: スコアリングジョブ]
                                                              |
                                                              v
                                                     data/scores/*.json
                                                       (git commit & push)
                                                              |
                                                              v
                                        [静的ホスティング] <--fetch-- [React(静的サイト)]
```

### 6.1 リポジトリ構成(案)
```
game-trend-dashboard/
├── .github/workflows/
│   ├── collect_frequent.yml   # Steam同接数・Twitch視聴者数(4時間毎)
│   └── collect_daily.yml      # Trends/YouTube/Reddit/ストアランキング/News + スコアリング(1日1回)
├── collector/                  # Python
│   ├── sources/
│   │   └── steam.py / twitch.py / trends.py / youtube.py / reddit.py / store_ranking.py
│   ├── scoring.py              # 正規化・水準/勢いスコア・Hot Words抽出
│   └── games_master.yml        # ゲームマスタ(手動管理、3.2のプラットフォームID紐付けを保持)
├── data/
│   ├── raw/YYYY-MM-DD.json     # 生データの日次スナップショット
│   └── scores/latest.json      # 最新の集計済みスコア(フロントが読む)
└── frontend/                   # React(Vite等)、ビルド成果物を静的ホスティングへデプロイ
```

### 6.2 GitHub Actionsワークフロー設計
- **collect_frequent.yml**: `cron: '0 */4 * * *'` でSteam同接数・Twitch視聴者数のみ取得し `data/raw/` に追記コミット
- **collect_daily.yml**: 1日1回(例: JST朝9時)実行。Trends・YouTube・Reddit・ストアランキング・News RSSを収集 → `collector/scoring.py` を実行して水準/勢いスコア・Hot Wordsを算出 → `data/scores/latest.json` と日次アーカイブを更新 → git commit & push
- APIキー等(Twitch Client ID/Secret、YouTube API Key、Reddit Client ID/Secret)は**GitHub Secrets**に格納し、ワークフロー内で環境変数として参照する
- 注意: GitHub Actionsのcronは負荷状況により実行が数分〜十数分遅延することがある(正確な時刻起動は保証されない)。日次バッチ設計であれば実用上問題にならない想定

### 6.3 データ永続化方式
SQLiteはバイナリファイルでgitとの差分管理と相性が悪い(コミットのたびにリポジトリが肥大化しやすい)ため、**生データ・集計結果ともJSON形式でコミットする**方式を採用する。

- `data/raw/`: 生データの日次スナップショット。肥大化を避けるため直近90日分のみ個別ファイルで保持し、それ以前は月次集計に丸めてアーカイブするローテーションルールを設ける(具体的な保持期間は運用開始後の実データ量を見て調整)
- `data/scores/latest.json`: フロントエンドが直接fetchする最新スコア。過去分もJSON配列として保持し、週次/月次/カスタム期間の集計はバックエンドAPIを持たないため**フロントエンド側(JS)でクライアントサイド集計**する

### 6.4 ホスティング先
GitHub Pagesの無料プランは、個人アカウントでは**privateリポジトリからのPages公開に対応していない**(公開するにはリポジトリをpublicにするか、GitHub Pro以上が必要)。APIキー類はGitHub Secretsで保護されリポジトリ本体には含まれないため公開して実害のある情報は基本的にない想定だが、この前提で良いか要確認。

- リポジトリをpublicにして良ければ → **GitHub Pages**が最も設定が簡単(同一リポジトリで完結)
- privateを維持したい場合 → **Cloudflare Pages**(無料枠でprivateリポジトリからのデプロイに対応)

特に指定がなければ、public + GitHub Pagesを既定として進める想定。

## 7. 可視化コンポーネント一覧

| コンポーネント | 用途 |
|---|---|
| ランキング表 + スパークライン | ホーム画面での一覧性重視の総合ビュー |
| 4象限マップ(水準×勢い) | 「定番」「絶好調」「急上昇中」「下火」を一目で分類 |
| バブルチャート(Steam同接×Twitch視聴者、サイズ=Trends) | プレイ人口と話題性のギャップを可視化 |
| バンプチャート(週次順位推移) | 中期的な順位変動 |
| レーダーチャート | ゲーム個別プロフィールカード(指標別の強み・弱み) |
| ヒートマップ(ゲーム×指標) | 多数のゲームを一覧でスキャン |
| カレンダーヒートマップ | 月間の話題密度 |
| 折れ線グラフ(複数ゲーム比較、期間選択可) | Google Trends比較ツール的な使い方 |
| Hot Wordsランキングリスト | 急上昇ワード(ワードクラウドは比較に不向きなため不採用、伸び率ソートのリスト形式を採用) |

## 8. 未決事項・今後の検討事項

- **ゲームマスタのID紐付け**: Steam AppID・ストアアプリID・Twitchゲーム名などを横断して同一ゲームと認識させるマッピング作業は手動運用になる見込み。新規ゲームを追跡対象に追加するたびに手作業が発生する
- **ストアランキング取得の技術的実現方法**: App Store/Google Playの公式APIが存在しないため、ランキングページの取得方法(スクレイピングの是非含む)を別途検討する必要がある
- **追跡対象ゲームの母集団選定基準**: 全ジャンル横断で追うか、特定ジャンル・プラットフォームに絞るかは未決定
- **ホスティング先**: ローカル運用のみか、どこかにデプロイするかは未決定
- **重み付けパラメータの初期値**: 水準/勢いの重み(50/50を仮置き)やカテゴリ内コアスコアの各指標重みは、実データを見ながらチューニングが必要
- **ランキング表の主キー未定**: カテゴリ内コアスコアと横断注目度スコア、どちらを7節のランキング表のデフォルトソート順にするか未決定
- **YouTube APIクォータと追跡ゲーム数の関係**: 無料枠は実質100検索/日程度(2.5参照)のため、追跡対象ゲーム数(1ゲームにつき日次収集+Hot Words用の追加検索で2回以上消費しうる)によっては上限に達する。母集団選定基準を決める際にこの制約を織り込む必要がある
- **リポジトリの公開範囲**: public前提(GitHub Pages利用)で進めて良いか、private維持(Cloudflare Pages利用)を望むか未確認(6.4参照)
